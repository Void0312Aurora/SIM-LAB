# 技术栈与系统架构

## 1. 文档目的

本文档用于明确项目第一阶段的推荐技术栈、系统边界、数据流、目录结构和实现顺序。

它的目标不是一次性决定所有未来细节，而是为第一版原型提供一个足够稳定、可扩展、便于迭代的工程基础。

本文档建立在以下共识之上：

- 首发版本以第三人称上帝视角、俯视角或 2.5D 为主
- 首发目标是“结构可信、模拟可信”，不是近距离写实
- 第一阶段重心是街区级自动生成、群体模拟和回放分析
- 第一人称沉浸式模式是后续扩展，而不是首发要求
- 当前更强调命令行友好、自动化友好和 agent 驱动调试

## 2. 架构结论

第一阶段推荐采用：

- 离线数据与规则流水线：`Python`
- 资产编译与程序化几何：`Blender + Python`
- 运行时模拟与可视化：`Godot`
- 运行时资产格式：`JSON + glTF`，必要时辅以自定义二进制缓存
- 结果记录与实验数据：`SQLite` 或 `JSONL`

一句话概括：

这个项目应被设计为“城市编译器 + 资产编译器 + 模拟播放器”，而不是单一的游戏工程。

也就是说：

- 在引擎外完成地图抓取、规范化、语义推导和几何编译
- 在引擎内完成加载、模拟、交互、回放和可视化

## 3. 为什么推荐 Python + Blender + Godot

### 3.1 Python 适合离线城市编译器

第一阶段最大的工程难点是 GIS 数据处理，而不是实时渲染。

Python 的优势在于：

- 地理数据生态成熟
- 便于做清洗、投影、图构建和规则推导
- 便于批量实验和快速验证
- 适合与机器学习和数据分析工具结合

建议重点依赖的 Python 生态：

- `geopandas`
- `shapely`
- `pyproj`
- `osmnx`
- `networkx`
- `rasterio`
- `pydantic`

### 3.2 Blender 适合资产编译器

`Blender` 当前不承担主运行时职责，但很适合承担：

- 建筑体块挤出
- 道路和地表 mesh 生成
- 程序化道具摆放
- 资产导出

它的优势在于：

- 命令行和 Python 自动化能力强
- 适合做 geometry processing
- 适合和 Python 数据流水线紧密配合
- 能在保持自动化的前提下提供少量人工精修入口

### 3.3 Godot 适合第一版运行时模拟器

第一阶段不追求超写实，而更需要：

- 快速搭建俯视角或 2.5D 运行时
- 便利地实现单位管理、UI、回放和调试工具
- 保留未来转向更复杂视觉表现的空间
- 支持命令行和 headless 驱动

Godot 的优势在于：

- 命令行和 headless 运行友好
- 文本化资源更适合自动化和版本控制
- 适合上帝视角模拟器
- 更贴近 agent 驱动和高频调试的工作流

### 3.4 为什么当前不优先选 Unreal 或纯 Web

`Unreal`：

- 强项在高保真表现
- 但第一阶段重点不在此
- 项目和团队负担更重

纯 `Web`：

- 适合地图分析和可视化
- 但在高密度实时群体模拟、复杂 3D 场景和后续游戏化扩展上会更受限

如果未来需要做网页版分析器，可以在离线编译器输出同一套中间数据，由 Web 工具复用。

## 4. 核心设计原则

### 4.1 运行时只消费“编译后资产”

不要让 Godot 在运行时直接解析杂乱的 OSM、DEM、街景或其他原始数据。

正确方式是：

- 原始公开数据进入离线编译器
- 编译器输出统一的运行时资产包
- Godot 只负责加载资产包并执行模拟

### 4.2 事实层和推导层分离

要明确区分：

- 哪些数据来自真实世界事实
- 哪些数据来自规则推导
- 哪些数据来自 ML 预测或人工修正

这样做的好处是：

- 便于追踪错误来源
- 便于替换数据源
- 便于未来做置信度管理和可视化审计

### 4.3 数据驱动优于硬编码

第一阶段很多内容都应采用数据驱动：

- 建筑用途映射
- NPC 行为参数
- 感染规则
- 封锁规则
- 场景事件配置

这样做可以极大提高调试和试验效率。

### 4.4 先做可复用中间层

第一版不应直接从“原始地图”一步到“引擎场景”。

建议保留清晰的中间产物：

- 规范化城市数据
- 推导后的语义图层
- 可供运行时加载的资产包

这会让我们后续更容易：

- 换地图
- 换规则
- 做批量实验
- 做 Web 分析工具
- 补第一人称模式

## 5. 总体架构

### 5.1 分层结构

建议拆成六层：

1. 数据接入层
2. 规范化与清洗层
3. 语义推导层
4. 几何与资产构建层
5. 运行时模拟层
6. 记录与分析层

### 5.2 数据流

整体数据流建议如下：

`公开数据 -> 原始缓存 -> 规范化城市数据 -> 语义推导 -> Blender 资产编译 -> 运行时资产包 -> Godot 加载 -> 模拟运行 -> 回放与统计输出`

### 5.3 模块边界

#### A. 数据接入层 `pipeline.ingest`

职责：

- 拉取或读取 OSM / Overture / DEM / population / 街景特征
- 保存原始文件或接口快照
- 记录来源、版本、许可、时间戳

输入：

- 外部公开数据

输出：

- 原始数据快照

#### B. 规范化层 `pipeline.normalize`

职责：

- 坐标投影统一
- 图层命名统一
- 修复基础拓扑问题
- 合并多源建筑和道路数据
- 将数据转成内部 canonical schema

输入：

- 原始数据快照

输出：

- 规范化城市数据包

#### C. 语义推导层 `pipeline.derive`

职责：

- 生成人行导航图与车辆导航图
- 生成建筑用途、入口候选点、封锁节点
- 生成安全区、疏散点、人口热图等模拟资产

输入：

- 规范化城市数据包

输出：

- 语义图层和模拟输入数据

#### D. 几何构建层 `pipeline.build` / `authoring.blender`

职责：

- 生成 LoD1 建筑体块
- 生成道路、人行区和路口 mesh
- 生成地表区块、围栏、基础道具占位
- 输出 glTF 或引擎可导入格式

输入：

- 规范化城市数据包
- 语义图层

输出：

- 运行时几何资产

#### E. 运行时模拟层 `runtime.godot`

职责：

- 加载运行时资产包
- 生成场景对象
- 管理时间推进
- 运行 NPC 行为、感染、封锁、事件
- 渲染 UI、热力图和调试图层

输入：

- 运行时资产包
- 场景配置

输出：

- 运行时状态
- 回放数据
- 统计结果

#### F. 记录与分析层 `runtime.recording` / `pipeline.analytics`

职责：

- 记录关键事件和单位轨迹
- 导出统计结果
- 支持单次回放和多次实验对比

输入：

- 运行时状态与事件流

输出：

- 回放文件
- 指标结果
- 实验对比数据

## 6. 数据契约建议

第一版应尽早定义统一的数据契约，否则后期会在管线和引擎之间不断返工。

推荐将运行时资产包拆成以下逻辑文件：

- `manifest.json`
- `world.json`
- `buildings.json`
- `zones.json`
- `nav_pedestrian.json`
- `nav_vehicle.json`
- `props.json`
- `scenario.json`
- `meshes/*.gltf`

## 7. 运行时系统设计

Godot 运行时建议拆成以下系统：

### 7.1 Scene Loader

负责：

- 加载资产包
- 生成地表、建筑和道具
- 注册导航数据

### 7.2 Simulation State

负责：

- 管理世界时间
- 管理实体集合
- 管理区域状态和全局事件

### 7.3 Actor System

负责：

- 平民、感染者、军队三类单位的更新
- 感知、目标选择、移动和状态切换

第一阶段建议先用：

- 数据驱动状态机

而不是一开始就上：

- 完整行为树
- GOAP

原因是第一版更需要可解释、可调试，而不是行为系统形式上的复杂。

### 7.4 Influence System

负责：

- 感染扩散
- 噪声刺激
- 危险热区
- 封锁状态

### 7.5 Replay & Metrics

负责：

- 记录单位轨迹
- 记录关键事件
- 统计感染、死亡、存活、疏散效率

### 7.6 Debug & Analysis UI

负责：

- 图层开关
- 轨迹回放
- 热力图查看
- 参数调试

## 8. 目录结构建议

建议仓库结构如下：

```text
urban-sim-lab/
  authoring/
    blender/
  configs/
  data/
  docs/
  pipeline/
    src/
      urban_sim_lab_pipeline/
  runtime/
    godot/
  scenarios/
  schemas/
```

各目录职责如下：

- `authoring/blender/`
  - Blender 自动化脚本与模板

- `configs/`
  - 全局配置、映射表、规则参数

- `data/`
  - 本地实验数据目录
  - 可再细分为 raw / normalized / runtime

- `docs/`
  - 产品、数据和架构文档

- `pipeline/`
  - Python 离线编译器
  - 包含 ingest / normalize / derive / build 等模块

- `runtime/godot/`
  - Godot 项目目录
  - 包含加载、模拟、UI 和回放系统

- `scenarios/`
  - 场景参数模板
  - 不同街区、不同事件设定的配置

- `schemas/`
  - 规范化中间数据和运行时资产包的 schema 定义

## 9. 第一阶段实现顺序

第一阶段不建议一开始同时推进全部模块。

推荐按以下顺序实现：

1. 定义 schema 和运行时资产包结构
2. 完成 OSM / Overture 的单街区导入与清洗
3. 生成建筑体块、道路和人行区域
4. 生成步行图、车辆图、入口候选点和安全区
5. 在 Godot 中加载资产包并渲染静态场景
6. 接入三类 NPC 的基础状态机与移动
7. 接入感染、恐慌、封锁和事件记录
8. 补回放、统计和热力图
9. 最后再做街具、视觉优化和局部 ML 补全

## 10. 第一版不建议过早做的事情

为了避免项目失控，第一阶段不建议优先做：

- 室内真实重建
- 第一人称完整交互
- 高精度立面重建
- 大规模枪战手感系统
- 全城级超大地图
- 多人联网
- 全自动高保真角色生成

这些都可能有价值，但不应进入第一阶段主路径。

## 11. 第一人称扩展路径

如果后续要补第一人称，推荐沿现有架构增量扩展，而不是推翻重来。

需要新增的主要模块包括：

- 更高保真几何与立面生成
- 室内模板系统
- 更高密度碰撞与可交互物
- 第一人称控制器与近距离感知逻辑
- 更细粒度的音频、遮挡和视觉系统

底层规范化数据、语义推导和场景配置应尽量保持复用。

## 12. 当前建议

第一阶段最重要的不是“选一个最强引擎”，而是建立一个足够清晰的资产编译边界。

当前推荐结论可以总结为三句话：

- 用 Python 处理真实世界数据和规则推导
- 用 Blender 承担程序化几何与资产编译
- 用 Godot 承担模拟、交互、可视化和回放
